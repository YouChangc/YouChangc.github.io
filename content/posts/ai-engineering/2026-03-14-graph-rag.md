图 RAG 可以概括成一件事：把知识存进图里，再从图结构中找相关实体、关系、路径或子图，转成 LLM 可用的上下文，最后生成答案。

它和常见的文本 RAG 不同的地方在于，重点不只是“找相关内容”，而是显式利用知识之间的连接关系。对于那些依赖实体关联、关系链路和局部结构的问题，图 RAG 会更自然。

## 图 RAG 的基本流程

一套典型的图 RAG 流程，通常可以拆成下面几个阶段。

### 1. 构图

先从原始数据里提取：

- 实体
- 关系
- 属性

这些内容会被组织成图结构，作为后续检索的基础。

### 2. 图索引

图建好之后，还需要做组织和索引，便于后续查询。常见形式包括：

- 节点索引
- 属性索引
- 社区划分
- 图嵌入
- 子图缓存

这一步的目的，是让图上的检索更高效。

### 3. 问题理解

收到用户问题之后，需要先把问题映射成图查询需求。例如：

- 找相关实体
- 找路径
- 找邻居
- 找某类关系
- 找某个社区或主题

只有先明确问题对应的图查询目标，后面的图检索才有方向。

### 4. 图检索

接着从图里找相关证据，常见检索对象包括：

- 相关节点
- 相关边
- k-hop 邻居
- 最短路径
- 子图
- 社区摘要

这一层是图 RAG 的核心，因为这里真正开始利用图结构。

### 5. 上下文构造

图检索得到的结果，通常还要转换成 LLM 可以直接理解的上下文，例如：

- 三元组列表
- 子图摘要
- 结构化说明
- 图和原始文档混合的上下文

这一步本质上是在把图结构“翻译”成模型可读的输入。

### 6. 生成答案

最后，LLM 基于图检索到的证据生成回答。

## 常见用法

图 RAG 不只有一种落地方式，常见有下面几类。

### 基于知识图谱的 Graph RAG

这类做法会直接构建知识图谱，并在图上完成数据组织和检索。

### 图 + 文本混合 Graph RAG

图不一定是唯一的数据载体，它也可以只是“骨架”。

常见做法是：

- 图里存实体和关系
- 每个节点或边再挂载原始文本证据
- 检索时先走图，再回到文本细节

例如，可以先找到一条实体关系链，再把对应的原始文档段落一并交给 LLM。

### Community-based Graph RAG

如果图比较大，可以先对大图做：

- 社区检测
- 聚类
- 分层摘要

然后在 query 到来以后：

- 先定位相关社区
- 用社区摘要做粗粒度召回
- 再细化到局部节点和边

### Graph + Embedding 检索

图检索也不一定只依赖图遍历，还可以结合向量。

做法通常是：

- 节点有 embedding
- query 先做 embedding
- 找最相关节点
- 再基于图结构扩展邻居或路径

## 常见检索方式

图 RAG 里的检索方式可以按粒度来理解。

### 节点检索

最基础的方式，是先找到种子节点：

1. 关键词匹配
2. 向量相似度匹配

### 邻居扩展

找到种子节点后，可以继续扩展：

- 1-hop 邻居
- 2-hop 邻居
- k-hop 子图

### 路径检索

路径检索关注的是两个实体之间的关系路径，例如：

- 最短路径
- 所有满足条件的关系链

### 子图检索

这种方式会直接提取围绕 query 的局部相关子图。

### 社区检索

这种方式会先定位到某个主题子图或社区。

### 图摘要检索

如果图太大，不一定直接给完整子图，也可以给摘要，例如：

- 局部摘要
- 社区摘要
- 多层级摘要

## LLM 怎么读图

图结构本身不是自然语言，所以通常要先转换表示形式。

### 三元组

```plain
APT29 --uses--> Domain_X
Domain_X --resolves_to--> IP_Y
IP_Y --hosted_by--> Provider_Z
```

### 路径描述

```plain
APT29 is linked to Provider_Z through Domain_X and IP_Y.
```

### 子图摘要

```plain
The relevant subgraph shows that APT29 uses Domain_X, which resolves to IP_Y. IP_Y is hosted by Provider_Z. This indicates an infrastructure chain connecting the organization to the hosting provider.
```

### 图 + 原文证据混合

```plain
• 先给关系链
• 再附上原文证据段落
```

## 小结

图 RAG 的关键，不只是把知识放到图里，而是从图结构中检索证据，再把这些证据整理成 LLM 可用的上下文。它特别适合那些需要围绕实体关系、路径链路和局部子图来回答的问题。
