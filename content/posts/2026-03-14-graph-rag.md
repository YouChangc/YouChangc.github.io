图 RAG 可以理解为：把知识存进图结构里，在回答问题时不只做文本召回，而是从图中检索实体、关系、路径或子图，再把这些结构化结果转换成 LLM 可用的上下文，最后生成答案。

和传统基于文本切片的 RAG 相比，图 RAG 更强调“关系”本身。它适合处理那些不仅需要找到相关内容，还需要解释实体之间如何连接的问题。

## 图 RAG 的基本流程

图 RAG 的流程可以拆成六步。

### 1. 构图

先从原始数据中提取：

- 实体
- 关系
- 属性

这些信息会被组织成图，为后续检索提供结构化基础。

### 2. 图索引

图建好之后，还需要做索引和组织，便于后续查询。常见方式包括：

- 节点索引
- 属性索引
- 社区划分
- 图嵌入
- 子图缓存

这一步的目标不是生成答案，而是让后面的检索更快、更准。

### 3. 问题理解

收到用户问题后，需要先把自然语言问题映射成图查询需求。常见映射方式包括：

- 找相关实体
- 找路径
- 找邻居
- 找某类关系
- 找某个社区或主题

这一步决定了后面会从图里召回什么样的结构。

### 4. 图检索

明确查询目标后，就可以开始从图中检索相关证据，例如：

- 相关节点
- 相关边
- k-hop 邻居
- 最短路径
- 子图
- 社区摘要

和纯向量召回不同，这里的核心是利用图结构把相关性展开。

### 5. 上下文构造

图检索得到的结果不能直接喂给 LLM，通常还需要转成模型容易消费的形式，例如：

- 三元组列表
- 子图摘要
- 结构化说明
- 图和原始文档混合的上下文

这一步本质上是在做“图到语言”的转换。

### 6. 生成答案

最后，LLM 基于图检索得到的证据生成回答。答案的质量取决于两个关键点：

- 图检索出的证据是否对问题足够相关
- 上下文表达是否足够清晰，让模型能正确理解结构关系

## 常见的图 RAG 形态

图 RAG 并不只有一种实现方式，常见有下面几类。

### 基于知识图谱的 Graph RAG

这是最直接的做法：构建知识图谱，把数据存在图里，再围绕图做检索。

它的重点是让实体和关系本身成为知识组织的中心。

### 图和文本混合的 Graph RAG

图不一定是唯一的数据载体，它也可以只是“骨架”。

一个常见做法是：

- 图里存实体和关系
- 在节点或边上挂载原始文本证据
- 检索时先走图，再回到文本细节

例如，先找到一条实体关系链，再把对应的原始文档段落一并提供给 LLM。这样既保留了图结构优势，也保留了文本细节。

### 基于社区的 Graph RAG

如果图很大，可以先做：

- 社区检测
- 聚类
- 分层摘要

当查询到来时，再：

- 先定位相关社区
- 用社区摘要做粗粒度召回
- 再细化到局部节点和边

这种方式适合在大图里分层处理问题。

### 图和 Embedding 结合的检索

图 RAG 也不一定只靠图遍历，还可以结合向量检索。

典型方式是：

- 节点带有 embedding
- query 先做 embedding
- 先找到最相关节点
- 再基于图结构扩展邻居或路径

这样可以同时利用语义相似度和结构关系。

## 常见图检索方式

在图 RAG 里，检索不只是一种动作，而是一组可以组合使用的方法。

### 节点检索

最基础的方式是先找到种子节点，常见方法有：

1. 关键词匹配
2. 向量相似度匹配

### 邻居扩展

找到种子节点后，可以继续扩展局部结构，例如：

- 1-hop 邻居
- 2-hop 邻居
- k-hop 子图

这类方法适合围绕某个实体补全上下文。

### 路径检索

路径检索关注的是“两个实体之间如何连接”。例如：

- 最短路径
- 所有满足条件的关系链

这类结果通常很适合回答因果链、依赖链、关联链这类问题。

### 子图检索

如果单个节点或路径不够，就可以直接提取围绕 query 的局部相关子图。

### 社区检索

当问题更偏主题或领域时，可以先定位某个主题子图或社区，再做更细的局部检索。

### 图摘要检索

如果图太大，不一定直接把完整子图提供给 LLM，也可以先提供摘要，例如：

- 局部摘要
- 社区摘要
- 多层级摘要

这样可以减少上下文长度，同时保留关键结构信息。

## LLM 怎么“读”图

图结构本身不是自然语言，所以要先转成模型容易理解的表示。

### 三元组

```plain
APT29 --uses--> Domain_X
Domain_X --resolves_to--> IP_Y
IP_Y --hosted_by--> Provider_Z
```

这是最直接的表达方式，保留了关系结构。

### 路径描述

```plain
APT29 is linked to Provider_Z through Domain_X and IP_Y.
```

这种形式更接近自然语言，适合把一条关系链压缩成一句话。

### 子图摘要

```plain
The relevant subgraph shows that APT29 uses Domain_X, which resolves to IP_Y. IP_Y is hosted by Provider_Z. This indicates an infrastructure chain connecting the organization to the hosting provider.
```

这种方式适合把局部结构浓缩成一段可读说明。

### 图和原文证据混合

还可以采用图和原始文本证据混合的方式：

- 先给关系链
- 再附上原文证据段落

这种方式通常更容易兼顾结构化关系和事实细节。

## 小结

图 RAG 的核心不只是“把知识放进图里”，而是把图结构真正用到检索和上下文构造中。它的价值主要体现在三个方面：

- 能显式利用实体之间的关系
- 能围绕路径、邻居、社区等结构做检索
- 能把图证据与原始文本结合，增强回答的可解释性

如果把传统 RAG 看成“从文本里找相关内容”，那么图 RAG 更像是“从结构化关系里找可推理的证据”。
